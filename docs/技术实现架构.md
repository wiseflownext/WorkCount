## 四、数据流架构

```
Screen (UI)
  │
  │ 1. 用户操作触发
  ▼
Store (Zustand)
  │
  │ 2. Store action 调用 Service
  ▼
Service 层
  │
  │ 3. Service 调用 Supabase SDK
  ▼
Supabase Client
  │
  │ 4. HTTPS 请求
  ▼
Supabase Cloud (PostgreSQL + RLS)
  │
  │ 5. 返回数据
  ▼
Service 层
  │
  │ 6. 数据转换 (snake_case → camelCase)
  ▼
Store (Zustand)
  │
  │ 7. 更新state，触发re-render
  ▼
Screen (UI 更新)
```

### 关键数据流示例

**计件录入流程**：
```
PieceEntryScreen
  → recordStore.addPieceRecord(productId, quantity)
    → record.service.addPieceRecord({
        user_id, product_id, quantity,
        unit_price_snapshot: product.unitPrice,  // 快照当前单价
        amount: quantity * product.unitPrice
      })
      → supabase.from('piece_records').insert(data)
    → recordStore 更新 pieceRecords
  → UI 显示成功
```

**计时打卡流程**：
```
TimeClockScreen
  → recordStore.startTimer(workContent, productId?)
    → record.service.startTimeRecord({
        user_id, work_content, product_id?,
        start_time: new Date(),
        hourly_rate_snapshot: currentUser.hourlyRate,
        status: 'working'
      })
      → supabase.from('time_records').insert(data)
    → recordStore 更新 activeTimeRecord
  → UI 进入计时状态 (setInterval 1000ms)

  → recordStore.endTimer(recordId)
    → record.service.endTimeRecord(id, {
        end_time: new Date(),
        duration_half_hours: Math.ceil(diffMinutes / 30),
        amount: halfHours * hourlyRateSnapshot,
        status: 'completed'
      })
      → supabase.from('time_records').update(data)
    → recordStore 更新，清除 activeTimeRecord
  → UI 显示结算结果
```

## 五、数据库设计

### 5.1 ER关系图

```
users (1) ──── (N) piece_records
  │                    │
  │                    └── product_id → products
  │
  └──── (N) time_records
               │
               └── product_id → products (nullable)

products (1) ──── (N) price_history
                         │
                         └── changed_by → users
```

### 5.2 表结构

```sql
-- 用户表
CREATE TABLE users (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  phone VARCHAR(11) UNIQUE NOT NULL,
  name VARCHAR(50) NOT NULL,
  role VARCHAR(10) NOT NULL DEFAULT 'worker'
    CHECK (role IN ('admin', 'worker')),
  hourly_rate DECIMAL(10,2) DEFAULT 0,
  status VARCHAR(10) DEFAULT 'active'
    CHECK (status IN ('active', 'inactive')),
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 产品工序表
CREATE TABLE products (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  code VARCHAR(20) UNIQUE NOT NULL,
  name VARCHAR(100) NOT NULL,
  type VARCHAR(10) NOT NULL
    CHECK (type IN ('product', 'process')),
  unit_price DECIMAL(10,2) NOT NULL,
  unit VARCHAR(10) DEFAULT '件',
  status VARCHAR(10) DEFAULT 'active'
    CHECK (status IN ('active', 'inactive')),
  remark TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 计件记录表
CREATE TABLE piece_records (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES users(id) NOT NULL,
  product_id UUID REFERENCES products(id) NOT NULL,
  product_name VARCHAR(100) NOT NULL,
  quantity INTEGER NOT NULL CHECK (quantity > 0),
  unit_price_snapshot DECIMAL(10,2) NOT NULL,
  amount DECIMAL(10,2) NOT NULL,
  work_date DATE NOT NULL DEFAULT CURRENT_DATE,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 计时记录表
CREATE TABLE time_records (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES users(id) NOT NULL,
  product_id UUID REFERENCES products(id),
  work_content VARCHAR(200) NOT NULL,
  start_time TIMESTAMPTZ NOT NULL,
  end_time TIMESTAMPTZ,
  duration_half_hours DECIMAL(5,1),
  hourly_rate_snapshot DECIMAL(10,2) NOT NULL,
  amount DECIMAL(10,2),
  work_date DATE NOT NULL DEFAULT CURRENT_DATE,
  status VARCHAR(10) DEFAULT 'working'
    CHECK (status IN ('working', 'completed')),
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 单价变更历史
CREATE TABLE price_history (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  product_id UUID REFERENCES products(id) NOT NULL,
  old_price DECIMAL(10,2) NOT NULL,
  new_price DECIMAL(10,2) NOT NULL,
  changed_by UUID REFERENCES users(id) NOT NULL,
  changed_at TIMESTAMPTZ DEFAULT NOW()
);
```

### 5.3 索引

```sql
CREATE INDEX idx_piece_user_date ON piece_records(user_id, work_date);
CREATE INDEX idx_time_user_date ON time_records(user_id, work_date);
CREATE INDEX idx_time_user_status ON time_records(user_id, status);
CREATE INDEX idx_price_product ON price_history(product_id, changed_at);
```

### 5.4 RLS策略

```sql
-- 员工只能读写自己的计件记录
CREATE POLICY piece_worker_select ON piece_records
  FOR SELECT USING (auth.uid() = user_id);
CREATE POLICY piece_worker_insert ON piece_records
  FOR INSERT WITH CHECK (auth.uid() = user_id);

-- 员工只能读写自己的计时记录
CREATE POLICY time_worker_select ON time_records
  FOR SELECT USING (auth.uid() = user_id);
CREATE POLICY time_worker_insert ON time_records
  FOR INSERT WITH CHECK (auth.uid() = user_id);
CREATE POLICY time_worker_update ON time_records
  FOR UPDATE USING (auth.uid() = user_id);

-- 员工只能看启用的产品
CREATE POLICY product_worker_select ON products
  FOR SELECT USING (status = 'active');

-- 管理员全权限
CREATE POLICY admin_all ON piece_records
  FOR ALL USING (
    EXISTS (SELECT 1 FROM users WHERE id = auth.uid() AND role = 'admin')
  );
-- (其余表同理)
```

## 七、核心业务规则

### 7.1 计件/计时互斥

```
录入计件前检查:
  SELECT id FROM time_records
  WHERE user_id = ? AND status = 'working'
  → 有记录 → 阻止，提示"请先结束计时"
  → 无记录 → 允许录入

开始计时前检查:
  SELECT id FROM time_records
  WHERE user_id = ? AND status = 'working'
  → 有记录 → 阻止，提示"已有进行中的计时"
  → 无记录 → 允许开始
```

### 7.2 单价快照机制

```
计件提交时:
  unit_price_snapshot = product.current_unit_price  (当前单价)
  amount = quantity × unit_price_snapshot
  → 后续product单价变更不影响此记录

计时开始时:
  hourly_rate_snapshot = user.current_hourly_rate  (当前时薪)
  → 后续user时薪变更不影响此记录

管理员修改单价时:
  → INSERT price_history (old_price, new_price)
  → UPDATE products SET unit_price = new_price
  → 只影响之后新建的记录
```

### 7.3 计时时长计算

```
结束计时时:
  diffMs = endTime - startTime
  diffMinutes = diffMs / 60000
  durationHalfHours = Math.ceil(diffMinutes / 30)  // 向上取整到半小时
  amount = durationHalfHours × hourlyRateSnapshot

示例:
  工作 45 分钟 → ceil(45/30) = 2 个半小时 → 2 × 时薪
  工作 20 分钟 → ceil(20/30) = 1 个半小时 → 1 × 时薪
  工作 60 分钟 → ceil(60/30) = 2 个半小时 → 2 × 时薪
```

## 十四、安全策略

| 措施 | 实现方式 |
|------|---------|
| 认证 | Supabase Auth JWT，自动刷新token |
| 数据隔离 | PostgreSQL RLS，员工只能访问自己数据 |
| 密码存储 | Supabase Auth内置bcrypt |
| 传输加密 | HTTPS (Supabase默认) |
| 会话管理 | AsyncStorage + 自动过期 |
| 敏感配置 | .env文件，不提交git |
| 管理员权限 | RLS + 应用层双重校验 |
